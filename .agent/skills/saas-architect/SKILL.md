# SaaS Architect — Next.js & Supabase

This skill governs all architectural decisions in this project. Follow every section strictly before writing or modifying any code.

---

## 1. Project Structure & Componentização

- **Atomic Design:** Decompose every complex UI into small, focused components inside `@/components`. One component = one responsibility.
- **Page-scoped components:** Components that belong exclusively to a specific page MUST be created inside a subfolder named after that page: `@/components/[page-name]/[ComponentName].tsx`. Only promote a component to the root `@/components/` level when it is reused across two or more pages.
- **shadcn/ui as base:** Use shadcn/ui as the default component library. Extend it (wrap, not fork) only when strictly necessary.
- **DRY:** If a data-fetching or formatting logic appears in more than one place, extract it to `@/lib` or a Custom Hook in `@/hooks`.
- **Route Groups (App Router):**
  - `app/(auth)/` — public authentication flows (login, signup, forgot-password).
  - `app/(protected)/` — all authenticated/protected application routes.
  - Each group has its own `layout.tsx` with appropriate middleware validation.

**Folder conventions:**
```
app/
  (auth)/
    login/page.tsx
    signup/page.tsx
  (protected)/
    dashboard/page.tsx
    layout.tsx          ← validates session server-side
components/
  ui/                   ← shadcn primitives (auto-generated, do not edit manually)
  [page-name]/          ← components used ONLY by that page (e.g. dashboard/, relatorio/)
    [ComponentName].tsx
  [shared-component].tsx ← promoted here only when reused by 2+ pages
lib/
  supabase/
    client.ts           ← browser client
    server.ts           ← server client (cookies)
  utils.ts
  validations.ts        ← Zod schemas (shared between client & server)
hooks/
  use-[name].ts
```

## 2. Segurança e Dados Sensíveis — Server-Side First

### 2.1 Zero Leakage
- **Never** import `SUPABASE_SERVICE_ROLE_KEY` or any secret in a client component or a file without `"use server"` / `server-only`.
- Only `NEXT_PUBLIC_*` variables may be referenced on the client.

### 2.2 Session Validation
- Always validate the session **on the server** using `supabase.auth.getUser()` inside Server Components or Server Actions.
- **Never** trust `supabase.auth.getSession()` alone on the server — `getUser()` re-validates with the Supabase API.
- Example pattern:

```ts
// app/(protected)/layout.tsx
import { createClient } from "@/lib/supabase/server";
import { redirect } from "next/navigation";

export default async function ProtectedLayout({ children }: { children: React.ReactNode }) {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect("/login");
  return <>{children}</>;
}
```

### 2.3 Server Actions — Mutação de Dados
- **All** create / update / delete operations MUST use Server Actions (`"use server"`).
- Every Server Action MUST validate its payload using a **Zod schema** before touching the database.
- Return typed results — never throw uncaught errors to the client.

```ts
// lib/validations.ts
import { z } from "zod";
export const CreateItemSchema = z.object({
  title: z.string().min(1).max(255),
  // ...
});

// app/(protected)/items/actions.ts
"use server";
import { createClient } from "@/lib/supabase/server";
import { CreateItemSchema } from "@/lib/validations";

export async function createItem(formData: FormData) {
  const parsed = CreateItemSchema.safeParse(Object.fromEntries(formData));
  if (!parsed.success) return { error: parsed.error.flatten() };
  const supabase = await createClient();
  const { error } = await supabase.from("items").insert(parsed.data);
  if (error) return { error: error.message };
  return { success: true };
}
```

### 2.4 RLS (Row Level Security)
- **Always** verify that RLS policies are enabled and correct for every table manipulated.
- Checklist before shipping any feature that touches a new table:
  - [ ] RLS enabled on the table in Supabase dashboard / migration.
  - [ ] `SELECT`, `INSERT`, `UPDATE`, `DELETE` policies narrowed to the authenticated user (`auth.uid()`).
  - [ ] Service-role operations (admin tasks) done only in Edge Functions or trusted server-side code — never exposed to the client.

## 3. Padrões de Código e UI

### 3.1 TypeScript Estrito
- `strict: true` in `tsconfig.json` — no exceptions.
- Use the types generated by Supabase CLI (`@/types/supabase.ts`) for all database entities:

```ts
import type { Database } from "@/types/supabase";
type Item = Database["public"]["Tables"]["items"]["Row"];
```

- Regenerate types after every schema migration:
  ```bash
  npx supabase gen types typescript --project-id <project-id> > types/supabase.ts
  ```

### 3.2 Feedback ao Usuário
Every async action MUST provide three states:

| State | Pattern |
|-------|---------|
| **Loading** | `<Skeleton />` from shadcn/ui while data fetches; `isPending` from `useTransition` during Server Action calls |
| **Error** | `toast.error(message)` via `sonner` |
| **Success** | `toast.success(message)` via `sonner` |

```tsx
"use client";
import { useTransition } from "react";
import { toast } from "sonner";
import { createItem } from "./actions";

export function CreateItemForm() {
  const [isPending, startTransition] = useTransition();

  function handleSubmit(formData: FormData) {
    startTransition(async () => {
      const result = await createItem(formData);
      if (result?.error) {
        toast.error("Algo deu errado. Tente novamente.");
      } else {
        toast.success("Item criado com sucesso!");
      }
    });
  }

  return (
    <form action={handleSubmit}>
      {/* ... */}
      <button disabled={isPending}>{isPending ? "Salvando…" : "Salvar"}</button>
    </form>
  );
}
```

### 3.3 Acessibilidade (a11y)
- Use shadcn/ui components — they are built on Radix UI and are ARIA-compliant by default.
- All interactive elements must be keyboard-navigable.
- Images must have descriptive `alt` attributes (or `alt=""` for decorative images).
- Color contrast must meet WCAG AA (4.5:1 for normal text).

### 3.4 Performance
- Fetch data in **Server Components** when possible; pass data as props to Client Components.
- Use URL `searchParams` for filter / pagination / tab state — ensures bookmarkable URLs.
- Use `next/image` for all images; `next/dynamic` with `ssr: false` for heavy client-only libraries.

## 4. Checklist de Review (aplique antes de finalizar qualquer PR)

- [ ] Nenhuma chave secreta exposta no cliente.
- [ ] RLS ativo e testado para as tabelas afetadas.
- [ ] Todas as mutações passam por Server Actions com validação Zod.
- [ ] Sessão validada via `getUser()` no servidor.
- [ ] Estados de loading, error e success implementados.
- [ ] TypeScript sem erros (nunca use `any` sem justificativa documentada).
- [ ] Componentes novos são pequenos, focados e reutilizáveis.
- [ ] Tipos do Supabase atualizados se houve mudança de schema.
